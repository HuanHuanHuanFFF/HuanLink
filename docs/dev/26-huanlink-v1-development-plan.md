# HuanLink v1.0 正式开发计划

> **状态：阶段级主计划。** 本文确定 Demo 之后的开发顺序、阶段边界和推进方式，不提前冻结各阶段内部实现。
>
> **For agentic workers:** 本文不能直接替代阶段设计和实施计划。进入每个里程碑前，必须先检查当时的代码与真实运行结果，和用户确认具体方案，再编写独立的 `Dxx` 实施计划。

**Goal:** 在已经跑通的真实 Demo 基础上，把 HuanLink 渐进发展为可日常使用、可配置、可扩展的本地单用户多 Agent 协作产品。

**Architecture:** 保留已经验证的 AgentCall、A2A、Codex app-server、Channel 和终态回流链路，按里程碑逐步替换 Demo 专用装配。MainAgent 负责外层理解与编排，各外部 Agent 通过标准协议或独立 Adapter 接入，具体执行能力不重新实现在 HuanLink Core 中。

**Tech Stack:** TypeScript、Node.js、pnpm workspace、OpenAI Agents JS、Vercel AI SDK、A2A Protocol v1.0、OneBot 11、Codex app-server。

## 2. 当前基线

当前 `main` 已经保留一条经过真实验收的闭环：

```text
真实 QQ
-> MainAgent
-> HuanLink AgentCall
-> 标准 A2A
-> Codex A2A Adapter
-> 官方 codex app-server
-> 真实代码修改与 Artifact
-> MainAgent 终态回流
-> 原 QQ 会话
```

已经可以复用的基础包括：

- AgentCall 的 `async` / `blocking`、查询、取消、输入补充和终态回流语义。
- A2A Agent Card、任务创建、订阅、查询、取消、状态和 Artifact 链路。
- Codex app-server 的进程管理、thread/turn 映射和真实代码执行。
- OneBot 11 WebSocket Channel、会话路由和原会话回复。
- EventLog、运行日志和真实链路关联 ID。

当前仍然带有明显 Demo 假设：

- 单项目、单 workspace、固定分支和环境变量配置。
- `apps/server` 仍使用 `phase3` / `phase4` 命名及固定 QQ、DeepSeek、Codex 装配。
- MainAgent 的工具和提示主要围绕 Codex 任务。
- 缺少面向用户的项目、工作目录、模型和 Agent 管理入口。
- 部分状态仍在内存中；v1.0 当前不要求服务重启后恢复。

## 3. 总体开发原则

### 3.1 渐进式替换，不全盘重写

不设置独立的“大重写阶段”。每个里程碑只重构阻碍本阶段目标的 Demo 边界，然后立即接入对应产品能力并进行真实复验。

```text
确认阶段目标
-> 保留现有行为测试
-> 替换必要的 Demo 装配
-> 实现本阶段能力
-> 局部测试和全链路复验
-> 报告结果并确认下一阶段
```

默认保留：

- Core 中已经稳定的协议无关契约与任务语义。
- A2A Client / Server 及 Codex app-server 通信。
- OneBot 11 Adapter、EventLog、运行日志和终态回流。
- 能准确描述当前行为的测试。

允许在相关阶段拆分或重构过大的实现文件，但不能仅因为文件较大就重新实现其已验证语义。

### 3.2 HuanLink 只拥有外层编排

HuanLink Core 负责 Channel、会话、MainAgent 调度、任务生命周期、AgentCall 路由和结果回流。单个 Agent 内部的推理、Tool Loop、代码修改或领域能力由相应框架或外部 Agent 自己负责。

不恢复旧自研通用 AgentLoop，也不创建接收任意 `toolName + args` 的万能异步网关。

### 3.3 配置所有权跟随运行边界

- MainAgent 使用的模型由 HuanLink 主服务独立配置。
- 外部 Agent 的执行配置由对应 Adapter 管理。
- Codex 的项目、workspace、工作目录、分支规则和执行模型属于 Codex A2A Adapter，不塞进 MainAgent 配置。
- HuanLink 只保存发现、选择和调用外部 Agent 所需的信息。

具体配置格式、存储、密钥引用和刷新方式在 M1 开始前另行确认。

### 3.4 v1.0 保持本地单用户边界

优先解决真实用户能否方便、清楚地使用，不提前建设多租户、组织权限、复杂持久化队列或工业化调度平台。只有真实使用暴露出明确问题时，才把相应基础设施加入后续计划。

## 4. M1：本地单用户配置化

### 目标

把当前固定环境 Demo 转换为可配置的本地单用户运行基线。

本阶段覆盖：

- 注册和选择项目。
- 选择项目内的工作目录作为工作焦点，而不是硬修改边界。
- MainAgent 模型与外部执行 Agent 模型独立配置。
- 为 MainAgent、Channel 和外部 Agent 建立清楚的配置所有权。
- 移除阻碍配置化的 `phase3` / `phase4` 和固定 Codex 装配，但不重写已验证核心。
- 保持当前 QQ -> Codex 真实闭环可运行。

### 进入本阶段前必须确认

- 配置文件格式和本地存放位置。
- 非敏感配置与密钥的分离方式。
- 配置变更何时生效。
- 项目、工作目录、分支和模型的选择语义。
- 同一 workspace 的最小并发规则。

### 完成标准

- 用户无需修改源代码即可切换已注册项目、工作目录和模型。
- MainAgent 模型选择不会隐式覆盖 Codex 等执行 Agent 的模型。
- Codex Adapter 能根据自己的配置验证目标项目与运行约束。
- 无效配置在启动或受理任务时给出明确错误。
- 原有真实 QQ、A2A、Codex、Artifact 和终态回流链路复验通过。

## 5. M2：MainAgent 编排能力

### 目标

让 MainAgent 不只会调用一个 Codex Tool，而是理解不同工作类型并合理安排执行。

本阶段覆盖：

- 区分直接回答、普通 Tool、Task-backed Tool 和子 Agent Task。
- 根据任务性质选择 `async` 或 `blocking`。
- 识别前后依赖，决定串行、并行、等待或向用户提问。
- 查询、取消、继续已有任务，而不是重复创建任务。
- 在终态回流时结合最新上下文决定回复、询问或继续已授权的下一步。
- 根据已注册 Agent 的能力选择执行者。

### 进入本阶段前必须确认

- Tool 和 Agent 能力如何描述给 MainAgent。
- 哪些判断交给模型，哪些约束由 HuanLink 机械保证。
- 多步骤任务的最小上下文和结果传递语义。
- 本阶段需要覆盖的真实编排场景。

### 完成标准

- MainAgent 能在确定性测试和真实场景中正确选择主要执行类型。
- 有依赖的步骤按顺序执行，互不依赖的工作可以并行且不阻塞其他会话。
- 异步任务受理、状态、输入补充和每个终态结果都能回到原会话。
- 不引入通用工作流 DSL、持久化状态机或新的自研 AgentLoop。

## 6. M3：配置驱动的模型与 Agent 扩展

### 目标

把“支持更多模型和 Agent”变成配置与适配能力，而不是在 Core 中逐个硬编码产品名称。

本阶段覆盖：

- 通过统一模型配置接入不同模型供应商，保持真实 `modelId` 与可选显示别名分离。
- 通过 Agent Card 和配置注册标准 A2A Agent。
- 对不支持 A2A 的平台使用独立 Adapter，不把其私有协议泄漏到 Core。
- 为后续 Channel 扩展保留同样的 Adapter 边界。
- 用新增配置验证 MainAgent 能发现、选择和调用不止一个能力来源。

### 进入本阶段前必须确认

- 模型供应商配置的统一层与供应商特有参数边界。
- A2A Agent 注册、能力刷新和认证信息边界。
- 用哪些新增模型和 Agent 作为本阶段真实验收对象。

### 完成标准

- 新增受支持模型时不需要修改 MainAgent 的业务编排代码。
- 新增标准 A2A Agent 时不需要修改 HuanLink Core。
- 非标准平台的适配代码被限制在独立 integration 或 Adapter 中。
- 至少一条新增配置驱动链路完成真实调用和结果回流验收。

## 7. M4：本地产品入口

### 目标

在配置和编排边界稳定后，为本地单用户提供清楚、可理解的日常使用入口。

本阶段覆盖：

- 管理项目、工作目录、MainAgent 模型和外部 Agent。
- 查看任务状态、结果、错误和运行日志。
- 简化首次配置、启动、停止和连接检查。
- 清楚展示模型原名、可选别名以及实际任务目标。

Web、桌面端、TUI 或其他交互形态不在本文中提前决定，进入 M4 前根据当时产品需求单独设计。

### 完成标准

- 用户不需要直接修改多处环境变量即可完成主要本地配置。
- 常见启动和连接错误可以从产品入口中定位。
- 用户能够确认当前会话、任务、项目、工作目录、模型和执行 Agent。
- 配置入口不绕过 M1 已建立的配置合同。

## 8. M5：v1.0 收敛与真实验收

### 目标

停止扩展新方向，集中验证 v1.0 已承诺的本地单用户体验。

本阶段覆盖：

- 使用多个项目、模型配置和 Agent 能力进行端到端验收。
- 验证普通 Tool、后台任务、子 Agent、状态查询和连续任务。
- 完善安装、配置、启动、故障定位和用户文档。
- 修复真实使用中出现的高频 P0 / P1 问题。
- 明确记录仍未支持的工业化能力。

### 完成标准

- 用户能够从一个受支持 Channel 发起任务并收到每个关键阶段的清楚反馈。
- MainAgent 能正确安排并追踪多个类型的工作。
- 外部 Agent 的实际结果、状态和错误能够回到原会话。
- 关键行为有自动化测试，核心产品路径完成真实验收。
- README 与正式文档只声明已经验证的能力。

## 9. 每个里程碑的执行门禁

每个里程碑均按下面顺序推进：

1. 检查当时的仓库、运行日志、依赖版本和未完成问题。
2. 和用户确认本阶段范围、非目标、关键语义和验收场景。
3. 编写并确认独立设计或 `Dxx` 实施计划。
4. 先补充能够保护现有行为的测试，再进行必要重构。
5. 按小批次实现，每个批次有独立验证结果。
6. 文档与代码分开提交，commit message 清楚说明范围。
7. 运行与风险相称的 package test、typecheck、build 和真实 smoke。
8. 报告实际结果与未解决问题，获得确认后再进入下一里程碑。

不能因为局部单元测试通过就宣称真实闭环完成，也不能为了清理代码扩大当前里程碑范围。

## 10. v1.0 暂缓项

除非真实使用证明必须提前解决，否则 v1.0 不建设：

- 多租户、组织级权限和远程公网鉴权平台。
- 服务重启后的完整任务与 Agent thread 恢复。
- 复杂持久化任务队列、分布式 exactly-once 和工作流 DSL。
- 文件级 ACL、`scopePaths` 和每任务独立 worktree。
- 自动 commit、push、merge 或部署。
- 通用递归子 Agent 创建框架。
- 自动模型路由、成本优化和供应商故障转移平台。
